# 앱 디자인 워크플로우

앱 기획부터 디자인, 구현까지의 전체 과정을 단계별로 안내하는 워크플로우이다.

## 실행 방법

사용자가 이 스킬을 호출하면 다음을 수행한다:

1. **현재 프로젝트 상태 파악** — 아래 진단 절차에 따라 어느 Phase에 있는지 판단
2. **다음 단계 안내** — 해당 Phase에서 해야 할 구체적 작업을 안내
3. **인자에 따른 동작 분기**:
   - 인자 없음 또는 `status`: 현재 상태 진단 및 다음 단계 안내
   - `next`: 바로 다음 해야 할 작업을 구체적으로 안내
   - `phase N`: 특정 Phase로 이동하여 해당 작업 안내

## 프로젝트 상태 진단 절차

다음 파일/디렉토리 존재 여부를 확인하여 현재 Phase를 판단한다:

```
1. 기획 문서 확인
   - docs/requirements*, docs/plans*, docs/service-overview* 등
   - README.md 에 프로젝트 개요가 있는지
   → 없으면: Phase 1 (기획) 필요

2. 디자인 시스템 확인
   - docs/design-system* 또는 유사 파일
   → 없으면: Phase 2 (디자인 시스템 정의) 필요

3. UI 스펙 확인
   - docs/ui-specs/*.md
   → 없거나 부족하면: Phase 3 (화면별 UI 스펙 생성) 필요

4. 디자인 파일 확인
   - *.pen 파일 존재 여부
   - designs/ 디렉토리
   → 없으면: Phase 4 (Pencil 디자인) 필요

5. 스펙 기반 구현 확인
   - specs/, changes/ 디렉토리
   - src/ 또는 lib/ 등 소스 코드
   → 없으면: Phase 5 (코드 구현) 필요

6. 모든 단계가 완료된 화면이 있고 미완료 화면이 남아있으면:
   → Phase 6 (다음 단계 반복)
```

## 전체 워크플로우

### Phase 1: 기획 (Planning)

**목표**: 프로젝트의 방향과 범위를 정한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- "이 프로젝트의 핵심 사용자는 누구인가요?"
- "MVP(최소 기능 제품)에 반드시 포함해야 할 기능은 무엇인가요?"
- "1단계로 먼저 만들고 싶은 화면이 있나요?"
- "참고하고 싶은 앱이나 서비스가 있나요?"

**수행 작업**:
- 기존 기획 문서 확인 (requirements, plans, service-overview 등)
- 부족한 부분을 사용자에게 질문하여 확인
- brainstorming 스킬로 아이디어 보완
- 구현 우선순위 확정 (단계별 범위 결정)
- 핵심: 모든 화면을 한번에 하지 않고, 단계별로 나눠서 진행

**산출물**:
- 기획 문서 (docs/requirements.md 또는 유사)
- 단계별 구현 계획 (어떤 화면을 먼저 할 것인지)

**완료 기준**:
- 프로젝트 목적과 핵심 기능이 문서화됨
- 1단계에서 구현할 화면 목록이 확정됨
- 사용자가 기획 방향에 동의함

---

### Phase 2: 디자인 시스템 정의

**목표**: 일관된 UI를 위한 디자인 시스템을 수립한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- "앱의 전체적인 느낌/분위기는 어떤 걸 원하시나요?" (선택지: 깔끔하고 미니멀 / 따뜻하고 친근한 / 전문적이고 신뢰감 있는 / 활기차고 에너지 넘치는)
- "선호하는 메인 색상 계열이 있나요?" (선택지: 블루 계열 / 그린 계열 / 오렌지/레드 계열 / 잘 모르겠다, 추천해주세요)
- "참고하고 싶은 앱의 디자인이 있나요?" (예: 토스, 배민, 카카오택시 등)
- "다크 모드를 지원해야 하나요?" (선택지: 라이트 모드만 / 다크 모드도 지원 / 시스템 설정 따르기)

**수행 작업**:
- 사용자의 디자인 방향 답변을 반영하여 디자인 시스템 생성
- UI UX Pro Max 스킬로 디자인 시스템 생성
  - 색상 팔레트 (Primary, Secondary, Neutral, Semantic 색상)
  - 타이포그래피 (폰트 패밀리, 크기 체계, 행간)
  - 간격 규칙 (spacing scale)
  - 기술 스택에 맞는 컴포넌트 패턴
- 앱의 성격에 맞는 톤앤매너 결정
- 생성된 디자인 시스템을 사용자에게 보여주고 피드백 반영

**산출물**:
- 디자인 시스템 문서 (docs/design-system.md)

**완료 기준**:
- 색상, 타이포그래피, 간격 규칙이 정의됨
- 주요 컴포넌트 패턴이 문서화됨
- 사용자가 디자인 방향에 동의함

---

### Phase 3: 화면별 UI 스펙 생성

**목표**: 현재 단계에서 구현할 화면들의 상세 UI 스펙을 작성한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- 각 화면별로 생성 전에 질문:
  - "이 화면에서 가장 중요한 정보/액션은 무엇인가요?"
  - "이 화면에서 특별히 원하는 레이아웃이나 배치가 있나요?"
  - "비슷한 화면을 가진 참고 앱이 있나요?"
- 생성 후 리뷰 요청:
  - "생성된 UI 스펙을 확인해주세요. 수정하고 싶은 부분이 있나요?"

**수행 작업**:
- 사용자와 화면별 요구사항 확인 후 스펙 생성
- ui-screen-spec 스킬로 현재 단계의 화면들에 대한 UI 스펙 생성
  - `/ui-screen-spec [screen-name]` 으로 개별 화면
  - `/ui-screen-spec all` 로 전체 화면
- 각 화면의 레이아웃, 컴포넌트 배치, 인터랙션 정의
- 화면 간 네비게이션 플로우 정의
- 생성된 스펙을 사용자에게 보여주고 피드백 반영

**산출물**:
- UI 스펙 문서들 (docs/ui-specs/*.md)

**완료 기준**:
- 현재 단계의 모든 화면에 대한 UI 스펙이 작성됨
- 각 스펙에 레이아웃, 컴포넌트, 데이터, 인터랙션이 포함됨
- 사용자가 각 화면 스펙을 승인함

---

### Phase 4: Pencil 디자인

**목표**: UI 스펙을 시각적 디자인으로 구현한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- 각 화면 디자인 후 스크린샷을 보여주며:
  - "이 디자인이 마음에 드시나요? 수정할 부분이 있나요?"
  - "레이아웃, 색상, 전체적인 느낌 중 변경하고 싶은 것이 있나요?"
- 전체 디자인 완료 후:
  - "전체 화면을 한번 살펴보셨나요? 일관성이나 흐름에서 아쉬운 점이 있나요?"

**수행 작업**:
- UI 스펙 문서를 기반으로 Pencil MCP 도구로 시각적 디자인
- 디자인 시스템 적용 (색상, 타이포그래피, 간격)
- 각 화면 완성 후 스크린샷으로 사용자에게 확인 요청
- 사용자 피드백 반영하여 수정
- 디자인 중 발견된 이슈는 기획/스펙 문서에 반영

**산출물**:
- Pencil 디자인 파일 (*.pen)
- 디자인 스크린샷

**완료 기준**:
- 현재 단계의 모든 화면이 디자인됨
- 디자인이 UI 스펙과 일치함
- 사용자가 디자인을 승인함
- 발견된 이슈가 기획/스펙에 반영됨

---

### Phase 5: 코드 구현

**목표**: 디자인을 실제 코드로 구현한다.

**수행 작업**:
- OpenSpec 도입하여 스펙 기반 코드 생성
  - `specs/` 에 현재 상태 기록
  - `changes/` 로 기능별 구현 관리
  - `/opsx:apply` 로 코드 생성
  - `/opsx:verify` 로 스펙 대비 검증
- 디자인 파일을 참조하여 픽셀 퍼펙트 구현
- 구현 후 검증 수행

**산출물**:
- 소스 코드
- 스펙 파일 (specs/, changes/)

**완료 기준**:
- 현재 단계의 모든 화면이 구현됨
- 스펙 대비 검증 통과
- 디자인과 구현이 일치함

---

### Phase 6: 다음 단계 반복

**목표**: 다음 단계 화면들에 대해 Phase 3~5를 반복한다.

**수행 작업**:
- 이전 단계에서 발견된 수정사항 반영
- 다음 단계 화면 목록 확인
- Phase 3 (UI 스펙) → Phase 4 (디자인) → Phase 5 (구현) 반복
- 필요시 디자인 시스템 업데이트

**완료 기준**:
- 모든 계획된 화면이 구현됨
- 전체 앱이 일관된 디자인과 동작을 보임

---

## 사용법

- `/app-design-workflow` — 현재 프로젝트 상태를 파악하고 다음 단계를 안내
- `/app-design-workflow status` — 현재 어느 Phase에 있는지 확인
- `/app-design-workflow next` — 다음 해야 할 작업 안내
- `/app-design-workflow phase 3` — 특정 Phase의 작업을 안내

## 필요한 스킬

| 스킬 | 용도 | 사용 Phase |
|------|------|-----------|
| brainstorming | 기획 보완, 아이디어 발산 | Phase 1 |
| ui-ux-pro-max | 디자인 시스템 생성 | Phase 2 |
| ui-screen-spec | 화면별 UI 스펙 작성 | Phase 3 |
| writing-plans | 계획 수립 및 문서화 | Phase 1, 6 |
| verification-before-completion | 완료 전 검증 | Phase 5 |

## 핵심 원칙

1. **사용자와 함께** — 모든 Phase에서 사용자에게 질문하고 피드백을 받는다. 일방적으로 결정하지 않는다. AskUserQuestion 도구를 적극 활용한다.
2. **점진적 진행** — 한번에 전체를 하지 않는다. 단계별로 나눠서 점진적으로 진행한다.
3. **완벽보다 진행** — 기획은 디자인하면서 구체화된다. 완벽한 기획을 기다리지 않는다.
4. **산출물 연결** — 각 Phase의 산출물이 다음 Phase의 입력이 된다.
5. **즉시 반영** — 디자인 중 발견된 이슈는 즉시 기획/스펙에 반영한다.
6. **검증 필수** — 구현 후에는 반드시 스펙 대비 검증을 수행한다.
7. **프로젝트 무관** — 이 워크플로우는 특정 프로젝트에 종속되지 않는다. 어떤 앱 프로젝트에도 동일하게 적용할 수 있다.
