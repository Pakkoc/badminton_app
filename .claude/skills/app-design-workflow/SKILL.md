# 앱 디자인 워크플로우

앱 기획부터 디자인, 구현까지의 전체 과정을 단계별로 안내하는 워크플로우이다.

## 실행 방법

사용자가 이 스킬을 호출하면 다음을 수행한다:

1. **현재 프로젝트 상태 파악** — 아래 진단 절차에 따라 어느 Phase에 있는지 판단
2. **다음 단계 안내** — 해당 Phase에서 해야 할 구체적 작업을 안내
3. **인자에 따른 동작 분기**:
   - 인자 없음 또는 `status`: 현재 상태 진단 및 다음 단계 안내
   - `next`: 바로 다음 해야 할 작업을 구체적으로 안내
   - `phase N`: 특정 Phase로 이동하여 해당 작업 안내

## 프로젝트 상태 진단 절차

다음 파일/디렉토리 존재 여부를 확인하여 현재 Phase를 판단한다:

```
0. 브레인스토밍 확인
   - 프로젝트 아이디어가 구체화되어 있는지
   - 핵심 기능과 사용자가 정의되어 있는지
   → 없으면: Phase 0 (브레인스토밍) 필요

1. 기획 문서 확인
   - docs/requirements*, docs/plans*, docs/service-overview* 등
   - README.md 에 프로젝트 개요가 있는지
   → 없으면: Phase 1 (기획) 필요

2. 기술 설계 확인
   - docs/techstack*
   - docs/database*
   → 없으면: Phase 2 (기술 설계) 필요

3. 유스케이스 확인
   - docs/usecases/**/*.md
   → 없거나 부족하면: Phase 3 (유스케이스) 필요
   ※ 유스케이스가 디자인의 기반이 된다. 반드시 디자인 전에 완료한다.

4. 디자인 시스템 확인
   - docs/design-system* 또는 유사 파일
   → 없으면: Phase 4 (디자인 시스템 정의) 필요

5. UI 스펙 확인
   - docs/ui-specs/*.md
   → 없거나 부족하면: Phase 5 (화면별 UI 스펙 생성) 필요

6. 디자인 파일 확인
   - *.pen 파일 존재 여부
   - designs/ 디렉토리
   → 없으면: Phase 6 (Pencil 디자인) 필요

7. 공통 모듈 확인
   - docs/common-modules*
   → 없으면: Phase 7 (공통 모듈 설계) 필요
   ※ 동기화 장벽: 이 단계 완료 전 화면별 구현 불가

8. 화면별 상태/로직 설계 확인
   - docs/pages/*/state.md
   - docs/pages/*/plan.md
   → 없거나 부족하면: Phase 8 (화면별 상태/로직 설계) 필요

9. 코드 구현 확인
   - src/ 또는 lib/ 등 소스 코드
   → 없으면: Phase 9 (구현) 필요

10. 구현 검증 확인
    - 테스트 통과 여부
    - 스펙 대비 검증
    → 미완료: Phase 10 (검증) 필요

11. 모든 단계가 완료된 화면이 있고 미완료 화면이 남아있으면:
    → Phase 11 (다음 단계 반복)
```

**정석 순서**: 기획 → 기술설계 → 유스케이스 → 디자인시스템 → UI스펙 → Pencil 디자인. 유스케이스가 "어떤 화면이 필요하고 어떤 흐름이어야 하는지"를 정의하므로, 디자인보다 반드시 먼저 완료한다.

**예외 (기존 프로젝트 편입)**: 디자인이 먼저 완성된 프로젝트는 Phase 2(기술설계) → Phase 3(유스케이스)를 보충한 후 Phase 7(공통 모듈)로 진행한다. 이 경우 유스케이스 작성 시 기존 디자인/UI스펙을 입력으로 활용한다.

## 전체 워크플로우

### Phase 0: 브레인스토밍 (Brainstorming)

**목표**: 프로젝트 아이디어를 발산하고 핵심 방향을 잡는다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- "어떤 앱/서비스를 만들고 싶으신가요?"
- "이 서비스가 해결하려는 문제는 무엇인가요?"
- "주요 사용자는 누구인가요?"
- "비슷한 서비스 중 참고하고 싶은 것이 있나요?"

**수행 작업**:
- brainstorming 스킬을 사용하여 아이디어 발산 및 구체화
- 사용자의 니즈와 문제 상황 탐색
- 핵심 기능과 차별점 도출
- 사용자 페르소나 정의
- 경쟁 서비스 분석 (있을 경우)
- 아이디어를 서비스 개요 문서로 정리

**산출물**:
- 서비스 개요 문서 (docs/service-overview.md)

**완료 기준**:
- 서비스의 핵심 가치와 해결하려는 문제가 명확함
- 주요 사용자와 사용 흐름이 정의됨
- 핵심 기능 목록이 도출됨
- 사용자가 서비스 방향에 동의함

---

### Phase 1: 기획 (Planning)

**목표**: 프로젝트의 방향과 범위를 정한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- "이 프로젝트의 핵심 사용자는 누구인가요?"
- "MVP에 반드시 포함해야 할 기능은 무엇인가요?"
- "1단계로 먼저 만들고 싶은 화면이 있나요?"
- "참고하고 싶은 앱이나 서비스가 있나요?"

**수행 작업**:
- 기존 기획 문서 확인 (requirements, plans, service-overview 등)
- 부족한 부분을 사용자에게 질문하여 확인
- brainstorming 스킬로 아이디어 보완
- 구현 우선순위 확정 (단계별 범위 결정)

**산출물**:
- 기획 문서 (docs/requirements.md 또는 유사)
- 단계별 구현 계획

**완료 기준**:
- 프로젝트 목적과 핵심 기능이 문서화됨
- 1단계에서 구현할 화면 목록이 확정됨
- 사용자가 기획 방향에 동의함

---

### Phase 2: 기술 설계 (Technical Design)

**목표**: 프로젝트의 기술 스택과 데이터베이스를 설계한다.

**수행 작업**:
- `/techstack` 스킬로 기술 스택 문서화
  - 프레임워크, 언어, DB, 인증, 외부 서비스
  - 각 기술 선택 이유
  - 버전 명시
- `/database` 스킬로 DB 스키마 설계
  - 데이터 흐름 서술
  - 테이블 정의 (컬럼, 타입, 제약조건)
  - 테이블 관계 (Mermaid ER 다이어그램)
  - RLS 정책, 인덱스, 마이그레이션 SQL

**산출물**:
- docs/techstack.md
- docs/database.md

**완료 기준**:
- 모든 기술 선택에 이유가 명시됨
- DB 스키마가 UI 스펙/기획의 데이터를 모두 커버함
- 과잉 설계 없이 필요한 테이블만 정의됨

---

### Phase 3: 유스케이스 (Use Cases)

**목표**: 기능 단위의 유스케이스를 작성한다.

**수행 작업**:
- `/usecase` 스킬로 기능별 유스케이스 작성
  - 기본 흐름, 대안 흐름, 예외 흐름
  - Mermaid 시퀀스 다이어그램
  - 에러 코드 정의
  - 테스트 시나리오

**산출물**:
- docs/usecases/{N}-{name}/spec.md (기능별)

**완료 기준**:
- 모든 핵심 기능에 대한 유스케이스가 작성됨
- 예외 흐름과 에러 코드가 정의됨
- 테스트 시나리오가 포함됨

---

### Phase 4: 디자인 시스템 정의

**목표**: 일관된 UI를 위한 디자인 시스템을 수립한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- "앱의 전체적인 느낌/분위기는 어떤 걸 원하시나요?"
- "선호하는 메인 색상 계열이 있나요?"
- "참고하고 싶은 앱의 디자인이 있나요?"
- "다크 모드를 지원해야 하나요?"

**수행 작업**:
- UI UX Pro Max 스킬로 디자인 시스템 생성
  - 색상 팔레트, 타이포그래피, 간격 규칙
  - 컴포넌트 패턴
- 톤앤매너 결정

**산출물**:
- docs/design-system.md

**완료 기준**:
- 색상, 타이포그래피, 간격 규칙이 정의됨
- 주요 컴포넌트 패턴이 문서화됨
- 사용자가 디자인 방향에 동의함

---

### Phase 5: 화면별 UI 스펙 생성

**목표**: 현재 단계에서 구현할 화면들의 상세 UI 스펙을 작성한다.

**수행 작업**:
- ui-screen-spec 스킬로 화면별 UI 스펙 생성
- 레이아웃, 컴포넌트 배치, 인터랙션 정의
- 화면 간 네비게이션 플로우 정의

**산출물**:
- docs/ui-specs/*.md

**완료 기준**:
- 현재 단계의 모든 화면에 대한 UI 스펙이 작성됨
- 사용자가 각 화면 스펙을 승인함

---

### Phase 6: Pencil 디자인

**목표**: UI 스펙을 시각적 디자인으로 구현한다.

**사용자에게 질문할 것** (AskUserQuestion 도구 사용):
- 각 화면 디자인 후 스크린샷을 보여주며:
  - "이 디자인이 마음에 드시나요? 수정할 부분이 있나요?"

**수행 작업**:
- UI 스펙 문서를 기반으로 Pencil MCP 도구로 시각적 디자인
- 디자인 시스템 적용
- 사용자 피드백 반영
- 디자인 중 발견된 이슈는 기획/스펙 문서에 반영

**수정 시 연관 화면 일관성 체크 (필수)**:

디자인 요소를 수정할 때마다 아래 절차를 반드시 수행한다:

1. **영향 범위 파악** — 수정하려는 요소가 다른 화면에도 존재하는지 확인
2. **일관성 대조** — 속성을 모든 연관 화면에서 비교
3. **일괄 수정** — 불일치 발견 시 모든 연관 화면을 한번에 수정
4. **스크린샷 검증** — 수정된 모든 화면의 스크린샷 확인

**공통 요소 체크리스트**:
- [ ] 지도 마커 (스타일, 크기, 아이콘)
- [ ] 상태 뱃지 (색상, 텍스트, 모양)
- [ ] 상태 전환 버튼 (디자인 시스템의 상태색 사용)
- [ ] 카드 컴포넌트 (모서리, 그림자, 패딩)
- [ ] 하단 네비게이션 (아이콘, 레이블, 활성 색상)
- [ ] 앱바 (높이, 폰트, 뒤로가기 아이콘)

**산출물**:
- Pencil 디자인 파일 (*.pen)

**완료 기준**:
- 현재 단계의 모든 화면이 디자인됨
- 디자인이 UI 스펙과 일치함
- 사용자가 디자인을 승인함

---

### Phase 7: 공통 모듈 설계

**목표**: 병렬 개발을 위한 공통 모듈을 설계한다.

⚠️ **동기화 장벽**: 이 Phase가 완료되기 전에 화면별 구현(Phase 9)을 시작하지 않는다.

**수행 작업**:
- `/common-modules` 스킬로 공통 모듈 설계
  - 초기화/설정, 인증, 라우팅, 에러 처리
  - 공통 모델/enum, 공통 위젯
  - 외부 서비스 래퍼, 유틸리티
  - 테스트 환경
- 3회 자가 검증 (완전성, 독립성, 최소성)

**산출물**:
- docs/common-modules.md

**완료 기준**:
- 모든 공통 요소가 도출됨
- 이 문서만으로 모든 화면을 병렬 개발 가능
- 3회 검증 통과

---

### Phase 8: 화면별 상태/로직 설계

**목표**: 각 화면의 내부 동작(상태 관리, 비즈니스 로직)과 구현 계획을 수립한다.

**수행 작업**:
- `/state-plan` 스킬로 화면별 상태 관리 설계
  - 상태/비-상태 데이터 분류
  - 상태 변화 조건표
  - Provider/Bloc 구조
  - 노출 인터페이스
- `/write-plan` 커맨드로 화면별 구현 계획 작성

**산출물**:
- docs/pages/{N}-{name}/state.md
- docs/pages/{N}-{name}/plan.md

**완료 기준**:
- 모든 화면에 상태 설계와 구현 계획이 있음
- 상태 변화 조건표가 완성됨

---

### Phase 9: 코드 구현

**목표**: 설계를 실제 코드로 구현한다.

**수행 작업**:
- `implementer` 에이전트를 사용하여 구현
  - 문서 전체 분석 → 체크리스트 생성 → 순차 구현 → 자가 검증 → 보고서
- Teams 환경에서 화면별 병렬 구현 가능
- TDD: 테스트 먼저 작성, 구현, 리팩토링
- 구현 중 설계 변경 발생 시 `/doc-sync` 실행

**산출물**:
- 소스 코드
- 단위 테스트

**완료 기준**:
- 현재 단계의 모든 화면이 구현됨
- 빌드/린트 에러 0
- 단위 테스트 통과

---

### Phase 10: 검증

**목표**: 구현이 설계대로 되었는지 검증한다.

**수행 작업**:
- `implement-checker` 에이전트로 스펙 준수 검증
  - 유스케이스 교차 검증 (기본/대안/예외 흐름)
  - 상태 관리 검증 (state.md 대조)
  - DB 스키마 검증 (database.md 대조)
- `code-reviewer` 에이전트로 코드 품질 검증
- `verification-before-completion` 스킬로 최종 확인

**산출물**:
- 검증 보고서
- 수정된 코드 (누락 발견 시)

**완료 기준**:
- 모든 유스케이스 흐름이 구현됨
- 상태 관리가 설계대로 동작함
- 코드 품질 기준 통과

---

### Phase 11: 다음 단계 반복

**목표**: 다음 단계 화면들에 대해 Phase 5~10을 반복한다.

**수행 작업**:
- 이전 단계에서 발견된 수정사항 반영
- 다음 단계 화면 목록 확인
- Phase 5 → 6 → 8 → 9 → 10 반복
- 필요시 디자인 시스템/공통 모듈 업데이트
- 변경 시 `/doc-sync`로 문서 동기화

**완료 기준**:
- 모든 계획된 화면이 구현됨
- 전체 앱이 일관된 디자인과 동작을 보임

### 기획 변경 대응 (Change Request)

프로젝트 진행 중 기획이 변경되는 경우 아래 절차를 따른다.

**절차:**

1. **영향 범위 분석**
   - 변경으로 인해 영향받는 화면/문서 목록 파악
   - 문서 의존 체인(techstack → database → usecases → common-modules → state-plan → plan) 참조
   - 영향 범위를 사용자에게 보고하고 확인

2. **설계 문서 갱신**
   - `/doc-sync` 스킬로 영향받는 문서를 상위부터 하위 순서로 갱신
   - 변경 이력을 docs/changes/에 기록

3. **산출물 동기화**
   - UI 스펙 문서 수정 → `/design-sync`로 Pencil 디자인 반영
   - 화면 추가/삭제 시 → `/sync`로 레지스트리 동기화

4. **검증**
   - 변경된 스펙과 디자인/코드가 일치하는지 확인

## 사용법

- `/app-design-workflow` — 현재 프로젝트 상태를 파악하고 다음 단계를 안내
- `/app-design-workflow status` — 현재 어느 Phase에 있는지 확인
- `/app-design-workflow next` — 다음 해야 할 작업 안내
- `/app-design-workflow phase 3` — 특정 Phase의 작업을 안내

## 핵심 원칙

1. **사용자와 함께** — 모든 Phase에서 사용자에게 질문하고 피드백을 받는다. AskUserQuestion 도구를 적극 활용한다.
2. **점진적 진행** — 한번에 전체를 하지 않는다. 단계별로 나눠서 점진적으로 진행한다.
3. **완벽보다 진행** — 기획은 디자인하면서 구체화된다. 완벽한 기획을 기다리지 않는다.
4. **산출물 연결** — 각 Phase의 산출물이 다음 Phase의 입력이 된다.
5. **즉시 반영** — 디자인 중 발견된 이슈는 즉시 기획/스펙에 반영한다.
6. **검증 필수** — 구현 후에는 반드시 스펙 대비 검증을 수행한다.
7. **프로젝트 무관** — 이 워크플로우는 어떤 앱 프로젝트에도 동일하게 적용할 수 있다.
8. **디자인 ↔ 스펙 동기화** — `/design-sync`로 Pencil→스펙, `/sync`로 레지스트리 정합성을 유지한다.
9. **수정 시 연관 화면 일괄 확인** — 한 화면의 디자인 요소를 수정할 때, 같은 요소가 존재하는 모든 화면을 찾아 함께 수정한다.
10. **문서 의존 체인** — 설계 변경 시 `/doc-sync`로 상위 문서부터 하위 문서까지 모두 갱신한다.
11. **동기화 장벽** — 공통 모듈(Phase 7) 완료 전에 화면별 구현을 시작하지 않는다.
