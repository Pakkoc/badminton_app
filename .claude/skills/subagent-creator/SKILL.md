# Subagent Creator

서브에이전트 정의 파일(`.claude/agents/{name}.md`)을 설계하고 생성하는 메타 스킬.

## 존재 이유

모든 에이전트는 일관된 구조(역할, 프로토콜, 검증, 제한사항)를 가져야 한다. 수동 생성은 필수 섹션 누락 위험이 있으므로, 이 스킬이 구조적 완전성을 자동으로 보장한다.

## 트리거

- "에이전트 만들어줘", "새 에이전트 생성", "agent 추가"
- "서브에이전트 만들어줘"
- "@{name} 에이전트 필요해"

---

## 생성 프로토콜

### Step 1: 요구사항 수집

사용자의 요청에서 다음을 파악:

| 항목 | 필수 | 설명 |
|------|------|------|
| **역할 (Role)** | ✅ | 에이전트가 수행할 핵심 작업 |
| **도메인 (Domain)** | ✅ | 전문 영역 (UX, 카피라이팅, 코드, QA 등) |
| **입출력 (I/O)** | ✅ | 무엇을 받아서 무엇을 산출하는가 |
| **상호작용 패턴** | ⬜ | 다른 에이전트와의 관계 (독립/의존/팀) |
| **특수 도구** | ⬜ | WebSearch, Bash 등 추가 도구 필요 여부 |

**질문 규칙**: 최대 3개 질문, 각 2-3개 선택지. 명확하면 질문 없이 진행.

### Step 2: 설계 결정

#### 2.1 모델 선택 (Model Selection Matrix)

| 기준 | opus | sonnet |
|------|------|--------|
| 복잡한 추론·분석·설계 | ✅ | |
| 창의적 생성 (카피, 디자인) | ✅ | |
| 적대적 리뷰·비판 | ✅ | |
| 기계적 검증·반복 작업 | | ✅ |
| 데이터 수집·정찰 | | ✅ |
| 포맷 변환·정리 | | ✅ |

> **기본값**: opus. sonnet은 창의적 판단이 불필요한 기계적 작업에만 사용.

#### 2.2 도구 선택 (최소 권한 원칙)

| 역할 유형 | 허용 도구 | 근거 |
|-----------|---------|------|
| **읽기 전용 분석** (리뷰, 감사) | Read, Glob, Grep | 산출물 오염 방지 |
| **생성·작성** (코드, 문서, 카피) | Read, Write, Edit, Glob, Grep | 파일 생성 필요 |
| **리서치** (조사, 벤치마크) | Read, Glob, Grep, WebSearch, WebFetch | 외부 정보 접근 |
| **구현·빌드** (코딩, 인프라) | Read, Write, Edit, Bash, Glob, Grep | 실행 환경 접근 |

> **Bash는 구현 에이전트에만.** 분석·리서치 에이전트에게 Bash를 주지 않는다.

#### 2.3 maxTurns 기준

| 작업 복잡도 | maxTurns | 예시 |
|------------|----------|------|
| 단순 (분석, 리뷰) | 15-20 | @design-reviewer |
| 표준 (생성, 구현) | 20-25 | @copywriter, @code-builder |
| 복잡 (리서치+생성) | 25-30 | @ux-researcher |

#### 2.4 메모리 설정

- **`memory: project`** — 세션 간 지식 축적이 필요한 에이전트 (기본값)
- **메모리 없음** — 일회성 또는 매번 백지에서 시작해야 하는 에이전트

### Step 3: 프로토콜 설계

에이전트의 작업 수행 절차를 설계:

1. **입력 단계**: 무엇을 읽는가 (소스 파일, 컨텍스트)
2. **처리 단계**: 핵심 작업 수행 절차
3. **자기 검증**: 산출물 품질 자가 점검
4. **출력 단계**: 산출물 형식 및 저장 위치

### Step 4: NEVER DO 정의

에이전트가 절대 하지 말아야 할 행동 목록:

- 역할 범위 외 작업 금지
- 도구 권한 외 행동 시도 금지
- 역할별 특화 금지사항 추가

### Step 5: 파일 생성

```
Write .claude/agents/{name}.md
```

**파일 구조** (references/agent-template.md 참조):

```markdown
---
name: {name}
description: {one-line description}
model: {opus|sonnet}
tools: {comma-separated tool list}
maxTurns: {15-30}
memory: project
---

{System prompt — role, absolute rules, protocol, NEVER DO}
```

### Step 6: 검증

생성된 에이전트 파일을 검증:

- [ ] Frontmatter 필수 필드 완전성 (name, description, model, tools, maxTurns)
- [ ] 도구 최소 권한 원칙 준수
- [ ] 모델 선택 근거 명확
- [ ] NEVER DO 섹션 존재
- [ ] 프로토콜 단계가 순서대로 정의

## 절대 기준

### 절대 기준 1: 최종 결과물의 품질
> 에이전트의 수를 줄여 비용을 아끼는 것보다, 전문 에이전트를 추가하여 품질을 높이는 것을 선택한다.

### 절대 기준 2: 최소 권한 원칙
> 에이전트에게 역할 수행에 필요한 최소한의 도구만 부여한다. "혹시 필요할까봐" 추가하지 않는다.

---

## 배치 생성 모드

여러 에이전트를 한 번에 생성할 때:

1. 전체 에이전트 목록 + 역할 정의를 먼저 작성
2. 에이전트 간 관계 (팀 구성, 의존성) 파악
3. 순서대로 생성 (의존 관계 순)
4. 각 에이전트 생성 후 즉시 검증
