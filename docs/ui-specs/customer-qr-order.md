# QR 접수 — UI 화면 스펙

> 최종 수정일: 2026-03-12

---

## 1. 화면 개요

| 항목 | 내용 |
|------|------|
| **화면 ID** | `customer-qr-order` |
| **화면 명** | QR 접수 |
| **목적** | 고객이 샵 QR코드를 스캔하면 자동으로 회원 등록 및 작업 접수를 처리하고, 결과를 안내한다 |
| **사용자 역할** | customer |
| **진입 조건** | 딥링크 `https://gutalarm.app/shop/{shopId}`로 진입 (QR 스캔) |

## 5. 사용자 인터랙션

| # | 이벤트 | 트리거 | 액션 | 결과 |
|---|--------|--------|------|------|
| 1 | 화면 진입 | 딥링크 수신 (QR 스캔) | 자동으로 회원 확인 → 등록 → 주문 생성 | 성공 시 완료 다이얼로그, 실패 시 에러 표시 |
| 2 | 접수 완료 확인 | "확인" 버튼 탭 | `/customer/home`으로 이동 | 고객 홈 화면 표시 |
| 3 | 에러 시 홈으로 | "홈으로" 버튼 탭 | `/customer/home`으로 이동 | 고객 홈 화면 표시 |
| 4 | 에러 시 로그인 | "로그인하기" 버튼 탭 | `/login`으로 이동 | 로그인 화면 표시 |

### 자동 처리 로직

```
딥링크 수신 (shop_id 추출)
  ↓
로그인 상태 확인
  ↓
├── 미로그인 → 에러: "로그인이 필요합니다"
│
└── 로그인 됨
      ↓
    샵 정보 조회
      ↓
    ├── 샵 없음 → 에러: "샵을 찾을 수 없습니다"
    │
    └── 샵 존재
          ↓
        members 테이블에서 회원 확인 (shop_id + user_id)
          ↓
        ├── 기존 회원 → 바로 주문 생성
        │
        └── 미등록 → 자동 회원 등록(members INSERT) → 주문 생성
              ↓
            orders INSERT (status = 'received')
              ↓
            접수 완료 다이얼로그 표시
```

## 6. API 연동

### 6.1 데이터 조회

| API | 테이블 | 조건 | 반환 데이터 | 호출 시점 |
|-----|--------|------|-------------|-----------|
| 사용자 정보 조회 | `users` | `id = 현재_사용자_id` | name, phone | 화면 진입 시 |
| 샵 정보 조회 | `shops` | `id = shop_id` | shop 레코드 | 화면 진입 시 |
| 회원 확인 | `members` | `shop_id + user_id` | member 레코드 또는 null | 화면 진입 시 |

### 6.2 데이터 변경

| API | 테이블 | 조건 | 변경 데이터 | 호출 시점 |
|-----|--------|------|-------------|-----------|
| 회원 자동 등록 | `members` | 해당 샵에 미등록 시 | shop_id, user_id, name, phone | 기존 회원이 없을 때 |
| 주문 생성 | `orders` | 없음 | shop_id, member_id, status='received' | 회원 확인/등록 후 |

### 6.3 실시간 구독

이 화면에서 실시간 구독은 사용하지 않는다.

## 10. 비고

- 이 화면은 Pencil 디자인이 없다 (자동 처리 화면이므로 별도 디자인 불필요).
- `ConsumerStatefulWidget`을 사용하며, `initState`에서 `Future.microtask`로 자동 처리를 시작한다.
- 회원 등록 시 freezed 모델의 빈 id(`''`)는 `MemberRepository.create()`에서 제거되어 DB가 UUID를 자동 생성한다.
- 주문 생성 후 DB 트리거 → Edge Function → FCM으로 고객에게 "접수됨" 푸시 알림이 전송된다.
- `barrierDismissible: false`로 설정하여 다이얼로그 바깥 터치로 닫히지 않도록 한다.

### 접근성

- AlertDialog의 아이콘과 텍스트가 접수 결과를 시각 + 텍스트로 전달하여 색상만으로 정보를 구분하지 않는다.
- 에러 메시지가 구체적이어서 사용자가 다음 행동을 알 수 있다.
